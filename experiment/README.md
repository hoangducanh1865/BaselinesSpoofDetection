# SEE-MoLEx explainability experiments

This directory explains the existing MoLEx, M2, and M3 checkpoints without
changing their parameters. The analysis is deliberately split into three
independent GPU suites.

## Updated interpretation

The two source corpora produce different interactions:

- With ASVspoof5 as source, M2 improves ASV19, ASV5, DFADD, and VSASV, but is
  worse than MoLEx on ITW, FoR, and VLSP. M3 is weaker than M2 on five of seven
  datasets.
- With ASVspoof2019 LA as source, M2 improves six of seven datasets and loses
  only on ITW. M3 improves over MoLEx everywhere and improves over M2 on five
  datasets, ties on VLSP, and loses on VSASV.

The experiments therefore test two separate claims:

1. Whether entropy-adaptive routing itself explains M2's domain-dependent gain.
2. Why the shared path complements M2 for ASV19 training but interferes for
   ASV5 training.

## Suite 1: native routing profile

`run_profiles.py` records per utterance and layer:

- normalized dense-router entropy, max probability, and top-1/top-2 margin;
- actual active count, determined from non-zero router weights rather than the
  fixed width of the returned index tensor;
- routed and total active counts separately for M1/M3, where the shared expert
  contributes one additional always-on path;
- active-count histograms, top expert IDs, utilization, gate mass, class
  utilization, and co-activation;
- utilization Gini/CV/dead-expert ratio;
- entropy AUROC for source-versus-OOD discrimination.

Attack labels are recovered for ASVspoof2019 LA and ASVspoof5 when their raw
protocols are available. VSASV/FoR/VLSP receive coarse path-derived groups.
Unknown metadata stays explicitly `unknown`.

## Suite 2: routing counterfactuals

`run_counterfactuals.py` performs the core 2x2:

| Weights | Evaluation policy |
|---|---|
| M0 | fixed top-4 |
| M0 | entropy-adaptive |
| M2 | fixed top-4 |
| M2 | entropy-adaptive |

It also evaluates M2 with fixed K=1..6, shuffled per-token active counts, and
random experts with the same active-count distribution. The latter two controls
separate "how many experts" from "the right experts for this token".

All comparisons use exactly the same deterministic stratified utterance sample.
Paired bootstrap confidence intervals are computed for EER differences.

## Suite 3: attribution

`run_attribution.py`:

- selects decision thresholds on source validation only;
- partitions target utterances into both-correct, rescued, harmed, and
  both-wrong groups for M0->M2 and M2->M3;
- compares entropy, active count, confidence margin, attack, and codec metadata
  between rescued and harmed samples;
- disables or scales the M3 shared/routed paths at inference;
- performs leave-one-routed-expert-out analysis for M2;
- computes paired bootstrap confidence intervals.

This distinction matters: EER measures score ranking, while rescued/harmed uses
a deployable threshold fixed before seeing the OOD target.

## Sampling and reproducibility

The default runs use deterministic, class-stratified subsets because evaluating
every policy on all 680k ASVspoof5 trials would be unnecessarily expensive.
Every artifact records the sample size, seed, checkpoint, source, dataset, and
policy. Use `--max-items 0` for a full-dataset run.

The diagnostic EERs produced here do not replace the full-dataset EERs in the
paper. They are paired mechanistic controls.

## Outputs

Artifacts are written below `experiment/results/` as CSV or compressed CSV:

- `profile/`: routing, utilization, co-activation, and OOD entropy (full
  utterance-layer records are sharded under `profile/details/`);
- `counterfactual/`: 2x2, fixed-K, shuffled-count, and random-expert controls;
- `attribution/`: thresholds, rescued/harmed, shared-path interventions,
  leave-one-expert-out, and bootstrap intervals;
- `summary.md`: combined report generated by `summarize.py`.

## Three-GPU launch

Run these in three terminals after activating the MoLEx environment:

```bash
CUDA_VISIBLE_DEVICES=0 python -u experiment/run_profiles.py \
  2>&1 | tee experiment/profile.log
```

```bash
CUDA_VISIBLE_DEVICES=1 python -u experiment/run_counterfactuals.py \
  2>&1 | tee experiment/counterfactual.log
```

```bash
CUDA_VISIBLE_DEVICES=2 python -u experiment/run_attribution.py \
  2>&1 | tee experiment/attribution.log
```

After all three finish:

```bash
python experiment/summarize.py --root experiment/results
```

## Optional fresh-training controls

Checkpoint-only counterfactuals cannot prove whether the two-epoch all-expert
warm-up caused the learned M2 representation. `train_controls.py` launches
fresh M2 controls for `default`, `no_warmup`, `max4`, and `static_tau`.
For a paper-level causal claim, run at least three seeds and select checkpoints
only by source dev EER.

Example:

```bash
CUDA_VISIBLE_DEVICES=0 python experiment/train_controls.py \
  --dataset asvspoof5 --control no_warmup --seed 1234
```
